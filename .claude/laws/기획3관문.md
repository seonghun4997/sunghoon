# 기획 3관문 — 새 기능·화면 착수 전 필독

## 관문 1: 텍스트 (착수 전 오너와)
- 저장소를 grep해 이미 있는 기능인지 먼저 확인한다. 있으면 재사용, 없을 때만 착수.
- 업무의 전 과정을 빠짐없이 펼친다. 예) 채용 = 지원→검토→과제→면접→합격→드랍→인재풀→온보딩.
- "이번에 만들 것 / 뺄 것 / 다음에 할 것"을 목록으로 보여주고 오너가 고른다.
- **빼기 질문 필수**: "이 중 뺄 것·합칠 것 있나요?"를 반드시 묻는다.

## 관문 2: 목업 (코드 전 오너와)
- 디자인·구조는 코드 전 2~3벌 병렬 제시 → 오너는 고른다(만들지 않는다).
- 오너가 고른 안만 구현한다. 나머지는 버린다.

## 관문 3: 개발 (오너 없이 — 오류를 오너에게서 차단)
- 오류를 오너가 먼저 발견하게 두지 않는다. 완료 선언 전 스스로 실행·확인한다.
- 오류 대응 순서 고정: ①원문·재현 확보 ②원인을 한 문장으로 확정 ③그 다음에만 수정.
- 원인을 못 쓰면 추측 수리 금지 — 로그·범위축소로 원인을 드러낸다. 그래도 모르면 오너에게 정확히 무엇이 필요한지 콕 집어 묻는다.

## 끝단: 판정 (오너와)
- 검증 통과한 결과만 오너에게 보인다.
- 오너는 방향·취향의 틀린 부분만 짚는다. 수정은 그 부분만.
- 디자인 N차는 정상 수렴 — 자책 유발 금지, 대신 한 바퀴를 싸게.

원칙: 오너에게 "만들게" 하지 말고 "고르거나·빼거나·짚게" 한다. 터치는 앞(지도)과 끝(판정)에만.

## 작업 대장 (BACKLOG.md)
- 루트의 BACKLOG.md가 이 프로젝트의 유일한 할 일 대장이다.
- 오너의 오더를 받으면: 코드 착수 전에 반드시 대장에 항목으로 먼저 기록한다 (오더 요약 + 날짜). 기획안에 기능이 5개면 5줄이다.
- 상태는 셋뿐: 🔲대기 / ⏸중단(어디까지 했고 뭐가 남았는지 한 줄 필수) / ✅완료(커밋 해시 병기)
- 매 세션 시작 시: BACKLOG.md를 먼저 읽고, 🔲·⏸가 있으면 "방치된 작업 N건: [목록]"을 첫 보고에 반드시 포함한다.
- 새 요청으로 하던 일을 끊을 때: 끊기는 작업을 ⏸로 기록한 뒤에만 전환.
- ✅ 전환은 검증 로그가 있을 때만.
- 오너가 "대장 보자"라고 하면 즉시 BACKLOG.md의 🔲·⏸ 전체를 보여준다.
