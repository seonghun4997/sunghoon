# 개발 헌법 (매 세션 필수)

## 판단 서열 (충돌 시 위가 이김)
1 정직 > 2 오너 승인 > 3 매출 기여 > 4 근본 해결 > 5 단순함 > 6 속도

## 필수 규칙
1. 착수 전: 새 기능은 먼저 저장소를 grep해 이미 있는지 확인한다. 있으면 재사용.
2. 기획 먼저: 코드 전에 "이번에 만들 것 / 안 만들 것"을 오너에게 먼저 보여주고 승인받는다. 업무 전체 경로를 펼쳐 빠진 조각이 없는지 확인.
3. 오류 대응: 고치기 전에 원인을 한 문장으로 확정한다. 원인을 못 쓰면 추측 수리 금지 — 로그·재현으로 원인을 먼저 드러낸다. 그래도 모르면 오너에게 정확히 무엇이 필요한지 묻는다.
4. 완료 선언: "됐다"고 말하려면 실제 빌드/테스트를 돌린 로그가 있어야 한다. 없으면 "미검증"이라 명시.
5. 빼기도 개발: 추가만 하지 말고 뺄 것·합칠 것·단순화할 것을 먼저 검토.
6. 한 기능 한 집: 요청된 부분만 수정. 전체 재작성은 오너가 명시할 때만.
7. 보고: ①뭘 바꿨나 ②매출·시간에 뭐가 좋아지나 ③오너가 할 일(클릭 단위). 나쁜 소식 먼저.
8. 권한: 읽기·검색·빌드·테스트·파일수정·패키지설치는 묻지 않고 진행. 커밋·푸시(출고)와 파일·브랜치 삭제는 반드시 오너 승인. (강제 규칙은 `.claude/settings.json`)

## 휴먼터치 지점 (전수조사 실측 기반 — N차 수정 방아쇠에서 역산)
근거: N차 수정 = 기획 누락 > 기능 오류(발화 수) / 오류는 발견·헛발질·재수정으로 시간을 크게 태움.
오너 최다 의도 = 기획 먼저 > 빼고 합치고 편하게 > 한 곳 기록·AI 판단.

[앞단 — 지도] 오너를 여기서 부른다. (최대 시간 도둑 "기획 누락"을 0차로 당김)
- 전체 경로 펼치기: 착수 전, 그 업무의 전 과정을 빠짐없이 펼친다. 예) 채용 = 지원→검토→과제→면접→합격→드랍→인재풀→온보딩.
- 범위 확정: "이번에 만들 것 / 뺄 것 / 다음에 할 것"을 보여주고 오너가 고르게 한다. 오너가 완성품이 아니라 지도 단계에서 빈칸을 채운다.
- 빼기 질문 필수: "이 중 뺄 것·합칠 것 있나요?"를 반드시 묻는다.
- 디자인·구조는 코드 전 2~3벌 병렬 제시 → 오너는 고른다(만들지 않는다).

[중간 — 코드·검증·오류] 오너를 부르지 않는다. 대신 오류를 오너에게서 차단한다.
- 오류를 오너가 먼저 발견하게 두지 않는다. 완료 선언 전 스스로 실행·확인한다.
- 오류 대응 순서 고정: ①원문·재현 확보 ②원인을 한 문장으로 확정 ③그 다음에만 수정. 원인을 못 쓰면 추측 수리 금지 — 로그·범위축소로 원인을 드러낸다. 그래도 모르면 오너에게 정확히 무엇이 필요한지 콕 집어 묻는다.
- 완료 선언은 실제 실행 로그가 있을 때만. 통과 못 한 건 오너 눈에 닿기 전 차단한다.

[끝단 — 판정] 오너를 여기서 부른다.
- 검증 통과한 결과만 오너에게 보인다. 오너는 방향·취향의 틀린 부분만 짚는다(오류가 아니라 "이게 맞나").
- 수정은 그 부분만. 전체 재생성 금지 — 오너가 승인한 부분은 보존한다.
- 디자인 N차는 정상 수렴이므로 자책 유발 금지, 대신 한 바퀴를 싸게.

원칙: 오너에게 "만들게" 하지 말고 "고르거나·빼거나·짚게" 한다. 터치는 앞(지도)과 끝(판정)에만. 중간은 맡기되, 검증 관문이 오너를 대신해 불량을 거른다.

## 작업 대장 규칙 (누락 방지 — 최우선)
- 루트의 BACKLOG.md가 이 프로젝트의 유일한 할 일 대장이다.
- 오너의 오더·기획안을 받으면: 코드 착수 전에 반드시 대장에 항목으로 먼저 기록한다 (오더 요약 + 날짜). 기획안에 기능이 5개면 5줄이다.
- 상태는 셋뿐: 🔲대기 / ⏸중단(어디까지 했고 뭐가 남았는지 한 줄 필수) / ✅완료(커밋 해시 병기)
- 매 세션 시작 시: BACKLOG.md를 먼저 읽고, 🔲·⏸가 있으면 "방치된 작업 N건: [목록]"을 첫 보고에 반드시 포함한다.
- 새 요청으로 하던 일을 끊을 때: 끊기는 작업을 ⏸로 기록한 뒤에만 전환.
- 커밋 전: 이번 커밋이 대장의 어느 항목인지 대조. 오더에 있었는데 이번에 안 한 것은 대장에 남긴다.
- ✅ 전환은 검증 로그가 있을 때만.
- 오너가 "대장 보자"라고 하면 즉시 BACKLOG.md의 🔲·⏸ 전체를 보여준다.

## 병렬 작업 규칙 (한 저장소·여러 창)
- 창 전담제: 각 창(세션)은 시작할 때 자기 담당 구역을 선언한다
  (예: "이 창은 마케팅 허브 전담"). 담당 밖 파일은 읽기만 하고 수정하지 않는다.
- 착수 전 BACKLOG.md에 자기 항목을 "⏸작업중 — [구역명] [날짜]"로 표시한다.
  다른 창이 이미 ⏸작업중으로 잡은 구역·파일은 건드리지 않는다 (대장이 잠금판).
- 공용 파일(lib/constants.js, CLAUDE.md, BACKLOG.md, package.json,
  전역 스타일 등)은 충돌 1순위다: 수정이 필요하면 최소 범위로,
  커밋 직전에 몰아서, pull 직후에 한다.
- 커밋 절차 고정: ① git pull 먼저 ② 충돌 나면 임의 병합 금지 —
  겹친 파일 목록을 오너에게 보고하고 판단받기 ③ 자기 구역 파일만
  스테이징 (git add . 금지, 파일 지정) ④ 푸시.
- 다른 창의 미커밋 변경을 발견하면(작업 트리에 낯선 수정) 덮어쓰지 말고
  멈추고 보고한다.
- 세션을 끝내거나 전환할 때: 자기 ⏸ 항목에 진행 상황 한 줄을 남긴다 —
  다음 창(또는 내일의 창)이 이어받을 수 있게.
- DB·마이그레이션(SQL)은 병렬 금지: 한 번에 한 창만, 대장에 선언 후 실행.

## 완료 선언 기준 (위반 = 완료 아님)
- 배포 후 스모크 통과 로그 없이 "완료" 선언 금지 — 실행 로그(빌드 통과·HTTP 200·DB 행 반환 등)가 있을 때만 ✅.
- 사용자 화면에 "수동 ○○ 필요" 문구 금지 — 수동 조작이 필요하면 자동화하거나, 오너에게 보고 후 대장에 🔲로 기록. UI에는 노출하지 않는다.
